## Skill trigger, ordering, and contract coherence check

The catalog now enforces primary order (1..4), finite `when` values, support-only targets, reciprocal `invokedBy`, and OpenCode command-template coherence. The static check I ran in this session (a one-off Node script that walks `catalog.yaml` directly) reports **0 issues** for: `mayInvoke` ⇄ `invokedBy` reciprocity, every support `mayInvoke` target covered by a trigger, every trigger target in the caller's `mayInvoke`, every trigger target in the target's `invokedBy`, every cross-reference resolving to an existing skill.

Reading the prose: every primary skill's SKILL.md now names the *trigger-based* invocation rule, not a free-form prose claim. For example, `skills/codepatrol-apply/SKILL.md:10` says "the catalog trigger `always-before-task-mutation` is the only path that invokes [execute-change]"; `skills/assess-change/SKILL.md:38` says "the catalog trigger `always-before-assessment` is the single path"; `skills/execute-change/SKILL.md:24` says the same. This is a substantive contract change vs. the pre-merge state: it removes the previous ambiguity where any primary could, in principle, name any support skill in prose and the reader had to trust the claim.

The *order* of primary skills is also now enforceable: lint rejects any primary whose `order` field is wrong, and rejects `codepatrol-status` for declaring any `order` at all. This eliminates a class of drift where a future harness adapter might re-arrange the primary skills.

**The one remaining prose-only claim that has not been converted to a machine-checkable rule:** the lifecycle stop rule. The four primary skills each have a "Stop rule (mandatory)" section saying "do NOT automatically invoke the next workflow". This is enforced by the prose contract, not by the CLI. A future improvement could add a CLI check that compares the actual call graph against the expected (e.g. `codepatrol-apply` should never spawn `codepatrol-verify`; recorded in a graph expectation file). This is a candidate improvement (option 3 below).

## Codepatrol-verify step re-evaluation

The apply session recorded three **minor documentation findings** in `verification.md`:

1. T6 journal entry in `implementation.md` is partially incomplete (`Result: pending` while the rest of the file documents the green T6 evidence). Non-blocker.
2. T3 journal entry has two `Result:` lines (the original `blocked` and the post-migration `blocked → resolved`). Honest snapshot; non-blocker.
3. `AGENTS.md` was listed in T5's plan file list but the diff shows no change. The verification confirmed AGENTS.md is already aligned with the contract, so no change was required. Plan drift; non-blocker.

The verifier classified all three as non-blockers because the *table-driven* acceptance evidence and the *final surface delta* are present and complete. The verifier-independence gap (same harness/model as apply) was also recorded, as in prior packages.

The verify step itself is **well-designed for its scope** (re-execute every AC, run `npm run verify`, audit blast radius, audit unplanned changes, classify findings). What the verify step **does not yet cover**, and what is a candidate improvement (option 2 below), is:

- **Cross-package dependency audit.** `graph impact --since-ref` enumerates files affected by this commit, but the verifier does not currently check whether any *governed package* (in `.codepatrol/packages/`) declares a dependency on a path that is now a behavioral change. For example: did the 2026-07-21-lean-docs package declare any contract that the T3 invariant would now break? Looking at it, the answer is "no" because lean-docs does not produce workflow items; but a generic check would be valuable.
- **Structural diff against the spec's `Expected surface delta` section.** The verify step audits the *actual* diff against the plan's task list, but the spec's `Expected surface delta` is checked only informally.
- **Re-execution of the bounded mechanical deviation's own acceptance test.** The verify step records the deviation as a journal entry but does not run a test that would fail if the deviation re-appeared (e.g. a test that scans the ledger for `nextAction: undefined` on non-closed items). This is the strongest test the project could add; it would have caught the original defect in this session.

## Market frameworks survey

The user requested a survey of relevant skills/frameworks. The point of this survey is to extract *concepts* that the project could adopt as a local pattern, not as a dependency. Per `skills/research-technology/SKILL.md`: "Never add the reference as a dependency, integration, adapter, plugin, compatible schema, or imported configuration by default."

Sources consulted (2026-07-21):

- [Agent Skills specification at `agentskills/agentskills` `38a2ff8…`](https://github.com/agentskills/agentskills/tree/38a2ff82958afee88dadf4831509e6f7e9d8ef4e). The spec defines `name`, `description`, and the optional `license`/`compatibility`/`metadata`/`allowed-tools` fields. Codepatrol already follows the required `name` + `description` and uses relative references one level deep.
- [OpenAI Agents SDK — handoffs and tracing](https://openai.github.io/openai-agents-python/handoffs/). The handoff-as-tuple and the per-agent guardrail pattern are useful, but Codepatrol already encodes handoffs through the artifact handoff contract and the trigger table. The per-agent guardrail pattern is the closest analogue to Codepatrol's `assertNextActionInvariant`; the project's runtime check is the next-level equivalent of an Agents SDK tool-guardrail.
- [MCP `2025-06-18` specification](https://github.com/modelcontextprotocol/modelcontextprotocol/blob/main/docs/specification/2025-06-18/index.mdx) and [MCP `2026-07-28` release candidate](https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/). The MCP transport-layer authorization model is useful as a reference for what a multi-tenant Codepatrol *could* do, but Codepatrol's product contract (`README.md`) is explicit: "The repository has no MCP server and no agent scheduler." MCP integration is therefore correctly out of scope.
- [AWS Bedrock AgentCore Memory SDK](https://github.com/aws/bedrock-agentcore-sdk-python). A managed session/memory service. Same scope decision: out of scope.
- [Microsoft `agent-skills`](https://github.com/microsoft/agent-skills) and [`jscraik/Agent-Skills`](https://github.com/jscraik/Agent-Skills). Both are upstream skill taxonomies with quality evals. Codepatrol already provides a *separate* skills tree under `skills/` and runs a per-skill lint; the eval arm is a candidate (option 1 below).
- [`Phoenixrr2113/agent-harness` — skill evals doc](https://github.com/Phoenixrr2113/agent-harness/blob/main/docs/skill-evals.md). Same lesson.
- MITRE ATLAS AI agent threat model. Out of scope for v1 but recorded as a future trigger for `security` and `authorization` cross-cutting concerns.
- [GitHub Copilot Coding Agent risks and mitigations](https://docs.github.com/en/copilot/concepts/agents/cloud-agent/risks-and-mitigations). Useful as a reference for "what can a coding agent do that this project must defend against" — and Codepatrol's design already implements most of the mitigations (review-gate, no automatic commits, no `subagent` tool, no MCP server, no agent scheduler, deterministic CLI gates).

**Synthesis.** The market has not produced a substantive *new* concept for a local, harness-agnostic, no-scheduler artifact-handoff tool. The skills specification, MCP, and the Agents SDK all target hosted or networked agent runtimes; Codepatrol's product contract deliberately excludes those. The *eval* discipline (`microsoft/agent-skills`, `Phoenixrr2113/agent-harness`, `jscraik/Agent-Skills`) is the single market concept that maps to a real gap in Codepatrol v1.0: the project has no executable evaluation harness for the skills. That is option 1 below.

## Candidate improvements — ranked

I evaluated the gap, the candidates, and the safety floor. Per `skills/solution-simplification/SKILL.md`: prefer the earlier sufficiency rung, and prefer reusing existing modules over new ones.

### 1. (Strong) Add an executable skill-evaluation harness (DC-1 from prior packages)

- **Files/seams:** new file `scripts/eval-skills.mjs` exporting `evalSkillTree(root)`, plus a new `lintSkillTree`-style contract that the catalog can re-use; a `docs/codepatrol/eval-skills/` bundle that records the red/green evidence per skill; the existing `pkg-contracts` can be extended with one more row.
- **Problem:** the project can lint the skill *structure* (frontmatter, links, triggers) but cannot test whether a skill actually behaves as advertised. Without evals, a skill that silently drifts in content is only caught when a plan invokes it and produces wrong evidence.
- **Proposed shape:** mirror `Phoenixrr2113/agent-harness/docs/skill-evals.md` — for each support skill, a small fixture that exercises the skill's invocation path against the local CLI; record the red/green result. Reuse the existing `lintSkillTree` export as the entry point. Do not introduce a new dependency; do not introduce a network service; do not introduce a hosted eval dashboard.
- **Wins:** leverage for callers (apply can prove the support skill it will invoke actually works); locality for maintainers (skill content drift is caught before apply); consistent with the project's "deterministic CLI gates" principle.
- **Risks:** the eval harness could become its own source of debt if it is over-engineered. Mitigated by writing one eval per skill at a time, only when a support skill's trigger is exercised in a real package.
- **Verification implications:** adds a new gate (`scripts/eval-skills.mjs`) that runs in `npm run verify`; failures are red-capable (a specific skill's eval is the gate, not the catalog as a whole).
- **Recommendation:** **Strong**. DC-1 has been a deferred constraint in multiple prior packages; this is the bounded upgrade path that DC-1's spec/plan records as the trigger condition.

### 2. (Strong) Tighten the `codepatrol-verify` step with a structural diff against the spec's `Expected surface delta`

- **Files/seams:** the `Verification Matrix` in `docs/workflow-memory.md` and the `codepatrol-verify` skill's `Re-verify independently` duty list. No new module; a new sub-routine in the existing `assess-change` skill.
- **Problem:** the verify step audits the actual diff against the plan's task list, but the spec's `Expected surface delta` is checked only informally. This was a *non-blocker* in the latest verify; it could be a blocker in a future package.
- **Proposed shape:** add a `compareSurfaceDelta(spec, gitDiff, allowList)` routine to `assess-change` that asserts: every file in the diff either is in the spec's expected surface delta, is in the bounded mechanical deviation list, or is the implementation/verification/handoff artifact of the package itself. Failures are reported as `implementation-defect` (unannounced file) and stop the verifier.
- **Wins:** closes a real evidence gap that the latest verify had to wave off as "non-blocker"; does not require a new dependency or harness; re-uses the existing `assess-change` skill.
- **Risks:** the surface delta is *expected* to be a reviewer's interpretation; over-asserting it could create false-positives. Mitigated by keeping the `allowList` for the package's own governance artifacts (handoff/implementation/verification/evidence) and by treating warnings rather than hard failures for "extras" outside the spec.
- **Verification implications:** adds a new pass in `codepatrol-verify`; the verify step itself becomes red-capable for surface-delta drift.
- **Recommendation:** **Strong**. Lower risk than (1) because it has no new executable surface; closes a documented gap.

### 3. (Worth exploring) Add a CLI check that the four primary skills' stop rules are not violated at runtime

- **Files/seams:** a new `scripts/check-lifecycle-invariance.mjs` that records an expected call graph (Plan → Review → Apply → Verify → none) and asserts against the catalog that no primary's `mayInvoke` reaches another primary except through `invoke` (not yet a field, would be a new addition).
- **Problem:** the stop-rule is enforced by prose, not by the toolchain. A future harness that imports Codepatrol skills but does not honor prose rules could silently chain Plan → Review → Apply → Verify in a single session, violating the contract.
- **Proposed shape:** the check would compare a recorded expectation (which primary may invoke which primary) against the runtime call graph when the package is in `review` or `apply` or `verify` state. Currently the spec forbids primary-to-primary invocation; the CLI could enforce it.
- **Wins:** closes the prose-only contract to a toolchain contract.
- **Risks:** speculative — no concrete incident has motivated it. Mitigated by treating the check as opt-in and recording the trigger as a deferred constraint.
- **Verification implications:** none directly, but the check would itself need a regression test.
- **Recommendation:** **Worth exploring, defer**. Not blocking; record as a deferred constraint with the same shape as DC-1 in prior packages.

### 4. (Speculative) Adopt an MCP server for multi-tenant Codepatrol

- **Files/seams:** new `src/mcp/` module; new harness adapter; new authorization and credential surface.
- **Problem:** none. The product contract is explicit: "The repository has no MCP server and no agent scheduler." No user or customer has asked for MCP integration.
- **Proposed shape:** none — reject. Re-evaluating this against the latest MCP spec does not change the product decision; if anything, the new MCP authorization model increases the surface area to defend.
- **Recommendation:** **reject**. Direct integration is correctly out of scope per `skills/research-technology/SKILL.md` (no default integration).

### 5. (Speculative) Adopt OpenAI Agents SDK tracing and guardrails

- **Files/seams:** new `src/tracing/` module; new instrumentation layer over the artifact handoff and workflow ledger.
- **Problem:** Codepatrol's product contract is local-only and the workflow ledger is already a structured event log. The Agents SDK tracing layer would add an external dependency and a hosted-tracing call path that contradicts the product contract.
- **Proposed shape:** the *concept* of structured per-task event records is already implemented in `implementation.md`. The concept of pre/post tool-guardrails is implemented in `assertNextActionInvariant` (T3) and in the `lintSkillTree` linter. Adopting the SDK as a runtime dependency is rejected; adopting the concept of "guardrails at the trigger seam" is already done.
- **Recommendation:** **reject** as direct integration; **adopted** as concept (already in place). No new code.

## Simplicity Decision

- **Selected rung:** reuse (option 2 — extend `assess-change` with a `compareSurfaceDelta` routine that the existing verify step can call). The eval harness (option 1) is a separate package because it is executable and adds a new gate; that belongs in its own work id.
- **Earlier rungs considered:** removing the verify step (rejected — its seven duties are necessary); collapsing the verify step into apply (rejected — the verifier-independence gap is the entire reason verify exists); deferring all of this (rejected — option 2 is a small, well-bounded extension with no new dependency).
- **Irreducible complexity:** option 2 adds one routine in `assess-change` and one assertion in the verify step. Option 1 adds one executable module and one new npm script. Both are small.
- **Safety floor:** preserved. No new dependency, no new public surface, no new harness, no new trust boundary. The diff will remain bounded and toolchain-readable.
- **Expected surface delta (option 2):** one new exported function in `assess-change`; one new call in the verify step; one new test in the contract suite. No new files, no new dependencies, no new npm scripts.
- **Expected surface delta (option 1, separate work id):** one new file `scripts/eval-skills.mjs`; one new npm script `eval:skills`; one new docs bundle. Optional `wiki` regeneration. No new dependencies, no hosted service.

## Risks and mitigations

- The two new patterns (`compareSurfaceDelta`, `eval-skills`) could each become a maintenance burden if they grow. Both are gated to one entry point and one new test, so the cost is bounded.
- Option 3 (lifecycle stop-rule check) is deferred. If a future incident shows the prose rule is fragile, the bounded upgrade is to record the call graph expectation and run a CLI check on it.
- The Qodo report re-evaluation found no missed item. The market-framework survey produced no substantive new concept for the local-only product. The only real gap is the eval discipline (option 1), which is the right next package to ship.
